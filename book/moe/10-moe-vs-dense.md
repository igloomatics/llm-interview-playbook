# 第 10 章 MoE vs Dense：什么时候用，什么时候别用

> 这一章不教你"怎么做 MoE"，教你"要不要做 MoE"。

---

## 10.1 一个常见的认知误区

很多人对 MoE 的第一印象是："**比 Dense 强**"。这是错的。

精确说法是：**给定相同激活参数，MoE 的总参数大、容量高，所以效果好；给定相同总参数，MoE 与 Dense 各有优劣**。

举例：

- **同激活**：Mixtral 8x7B (13B 激活) vs Llama2-13B → Mixtral 略胜（因为容量是 47B vs 13B）；
- **同总参**：Mixtral 8x7B (47B) vs Llama2-70B → 后者略胜（70B Dense 容量更高）；
- **同算力**：MoE 训练更省 FLOPs，但需要更多显存与通信带宽。

所以选 MoE 还是 Dense，要看你的**约束**是什么。

---

## 10.2 MoE 的优势

### 10.2.1 同算力下质量更高

GLaM 论文证明：**相同 FLOPs**，MoE 的 zero-shot 评测平均高 5.8 分。这是 MoE 的核心卖点。

### 10.2.2 推理时激活参数小

Mixtral 8x7B 推理 FLOPs ≈ Llama 13B，但质量接近 Llama 70B。**对推理成本敏感的场景**（API 服务、SaaS），MoE 直接省成本。

### 10.2.3 训练时间更省

GLaM：训练 1.2T MoE 比 GPT-3 (175B Dense) 省 65% 能耗。

### 10.2.4 容量上限更高

万亿参数 Dense 在 2025 年仍是工程极限（PaLM 540B 是当时极限）。MoE 已经做到 10T+（Switch 2048 expert × 几 B 每专家）。

### 10.2.5 多任务 / 多语言天然合适

不同专家可以学不同领域 / 语言。GShard 在 100 种语言翻译上取得了 Dense 远不能及的效果。

---

## 10.3 MoE 的劣势

### 10.3.1 显存压力大

**激活参数小不等于显存小**。所有专家都得驻留：

- Mixtral 8x7B：47B 参数 × 2 bytes = 94 GB → **单 H100 装不下**；
- Llama 13B：13B × 2 = 26 GB → 单 H100 轻松。

### 10.3.2 训练复杂度高

- All-to-All 通信；
- 负载均衡损失调参；
- 数值稳定性（Z-Loss）；
- 路由器训练不稳；
- 单卡 OOM 风险。

### 10.3.3 微调困难

- 容易过拟合（参数多、激活少）；
- 路由器在小数据上学不动；
- 需要"是否 freeze router"等额外决策。

### 10.3.4 推理基础设施门槛高

- 需要 fused MoE kernel；
- 需要 EP 多卡部署或量化；
- vLLM、SGLang 等是必需品；
- 边缘部署比 Dense 难。

### 10.3.5 质量上限可能不如同总参 Dense

Dense 70B 在某些任务上比 MoE 47B 更好。**总参数仍然是 capability ceiling 的主要决定因素**，激活参数决定的是 efficiency。

---

## 10.4 一个简洁的决策框架

按你的核心约束，做下面的选择：

```
Q1: 你训练算力受限吗？
    Yes  → MoE（同 FLOPs 质量更高）
    No   → 看 Q2

Q2: 你推理对延迟 / 成本敏感吗？
    Yes  → MoE（激活参数小，推理便宜）
    No   → 看 Q3

Q3: 你部署显存有限吗？
    Yes  → Dense（显存利用率高）
    No   → 看 Q4

Q4: 你需要做 finetune / RLHF / 持续训练吗？
    Yes (小数据)  → Dense（finetune 简单）
    Yes (大数据)  → MoE（仍然受益）

Q5: 你需要长上下文（>32k）吗？
    Yes  → 看模型支持 MLA/GQA：MoE+MLA > Dense+GQA > MoE+MHA
    No   → 主要看 Q1–Q3
```

## 10.5 真实业务场景对照

### 10.5.1 SaaS API 服务（OpenAI、Anthropic 风格）

- 高并发、低延迟需求；
- 显存可堆 → 优先 MoE；
- 推理总成本 = 激活参数 × token 数 → MoE 直接省钱；
- **GPT-4 / Gemini 1.5 / Claude 4 都是 MoE**（公开或推测）。

### 10.5.2 私有部署 / 企业本地

- 显存有限（4×A100 / H100 是常见）；
- 不需要极致延迟；
- Dense 70B 量级 + GQA，足以满足；
- MoE 部署门槛过高，**通常不是首选**。

### 10.5.3 边缘 / 移动端

- 显存极度有限（4–16 GB）；
- 需要 4bit / 量化；
- Dense 7B / 13B 是主流；
- MoE 在端上几乎不可行（最近有 Phi-3.5-MoE 这样的尝试，但仍小众）。

### 10.5.4 科研 / 探索

- 显存通常充足（多卡集群）；
- 想做大规模实验；
- MoE 是当前学术热点 → 推荐尝试。

### 10.5.5 finetune 现成模型

- 小数据集（<10k 样本）：选 Dense；
- 大数据集（>1M）：MoE 仍可，但要注意 freeze router；
- 极小数据 + LoRA：选 Dense（MoE LoRA 仍是研究方向）。

---

## 10.6 一个被忽视的因素：**生态成熟度**

到 2026 年初，Dense 的生态比 MoE 成熟很多：

| 维度 | Dense | MoE |
|------|-------|-----|
| 推理框架 | vLLM/llama.cpp/SGLang/TGI 全支持 | 大模型框架支持，小框架支持差 |
| 量化工具 | GPTQ/AWQ/GGUF 全支持 | 量化效果与稳定性参差 |
| LoRA 微调 | 成熟 | 路由器 LoRA 仍是研究 |
| 移动端 | MLX/CoreML/llama.cpp 全支持 | 几乎没有 |
| 推理库教程 | 丰富 | 偏论文化 |

**实务建议**：如果你的团队工程力量有限，且业务对推理基础设施敏感，**默认选 Dense，除非有明确理由用 MoE**。

---

## 10.7 一组"如果……就……"的实战经验

- 如果你**只是要做产品并 finetune**：默认选 Dense（Llama 3 8B/70B、Qwen 2.5 7B/72B）；
- 如果你要做**自家大模型 pretraining**：选 MoE（DeepSeek-MoE 风格）；
- 如果你要做**多语言 / 多领域**：选 MoE，专家天然分工；
- 如果你要做**边缘部署**：选 Dense + 量化；
- 如果你要做**长上下文**：选 MoE + MLA / Dense + GQA，看预算；
- 如果你要做**研究**：MoE 在 2024–2026 年都是热点方向。

## 10.8 几个常被问错的问题

**Q1：MoE 的总参数代表什么？**
A：它是模型的"知识容量"，但单步推理用不到全部。把它理解为"图书馆的书架数量"，激活参数才是"今天读的书数量"。

**Q2：MoE 是不是 Ensemble？**
A：不是。Ensemble 各模型独立训练、独立推理，MoE 端到端联合训练，单次推理只激活一部分。详见第 2 章。

**Q3：推理 MoE 比 Dense 慢吗？**
A：相同激活参数下相当或略慢（因为有路由 + All-to-All）。但 MoE 的"激活参数"通常比同质量的 Dense 小很多，所以**实际部署中 MoE 推理常常更快**。

**Q4：MoE finetune 是不是不行？**
A：不是不行，是更难调。小数据集 finetune Dense 更省心；大数据集 / 持续 pretraining，MoE 仍受益。

**Q5：MoE 是大模型未来吗？**
A：在算力受限的前沿大模型上，几乎是。但 7B 量级的小模型仍以 Dense 为主——MoE 只在足够大才划算。

## 10.10 本章要点

- 选 MoE 还是 Dense，看你的**核心约束**：算力、推理成本、显存、微调、上下文；
- MoE 在**算力受限的大模型 pretraining + 高并发推理**最有优势；
- MoE 在**小数据 finetune、边缘部署、显存极限**上有劣势；
- 生态成熟度仍偏向 Dense，**没明确理由就别上 MoE**；
- "MoE 一定比 Dense 强" 是错觉——它们是工程 trade-off 的两个端点。
